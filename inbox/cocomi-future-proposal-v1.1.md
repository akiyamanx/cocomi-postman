## 1. エグゼクティブサマリー

本企画書は、COCOMI OSの次フェーズとして計画する4つの主要構想をまとめたもの。
いずれも「語りコード（KatariCode）」のビジョン——「Tell, and it's built」——を一般ユーザーに展開するための基盤技術。

- **構想1:** 汎用APIオンボーディングモジュール → APIキー取得の難しさを解消
- **構想2:** LINEリッチメニュー×対話フロー開発ナビ → typoリスクゼロ、直感的開発指示
- **構想3:** Postmanアイデアカテゴリ・外部記憶システム → AIメモリ解放
- **構想4:** セットアップインストーラー構想 → Postman導入障壁の根本解決 ⭐NEW

---

## 2. 背景と課題

### 2.1 COCOMI OSの現状
COCOMI OSは「語りコード（KatariCode）」をコアコンセプトとし、音声入力×AIでフルコードを生成する開発方式を確立。現在、Postman Phase2b全自動フロー（LINE→Worker→GitHub→Claude Code→CI→LINE）が稼働中で、7つのリポジトリを統一CI/CDで運用中。

### 2.2 解決すべき課題

**課題1: APIキー取得の難しさ**
AI APIを使うアプリを一般ユーザーに展開する際、最大の障壁が「APIキーの取得」。従来フローでは推定8割のユーザーが途中で脱落する。

| ステップ | 従来の操作 | 脱落リスク |
|---------|-----------|----------|
| 1 | Google AI Studioを開く | 低 |
| 2 | Googleアカウントでログイン | 低 |
| 3 | 「Get API Key」を探す | 中 |
| 4 | プロジェクトを作成 | 高 |
| 5 | キーをコピー | 中 |
| 6 | どこに貼ればいいかわからない... | 最高 |

**課題2: PostmanのtypoリスクＡ**
現在のPostmanはテキスト入力ベースのため、プロジェクト名やコマンドの手打ちにtypoリスクが伴う。安全バリデーションで誤認識を防いでいるが、そもそも手打ち自体を不要にするUIXが理想。

**課題3: AIメモリの圧迫**
構想やアイデアをAIのメモリに載せたままだと、コンテキストウィンドウを圧迫し、実作業に使える容量が減る。人間がメモ帳に書き出して頭をクリアにするように、AIにも「外部記憶装置」が必要。

**課題4: Postman導入の技術的障壁 ⭐NEW**
Postmanの利用には以下の技術スタックが全て必要：
- LINE公式アカウント（Messaging API）
- Cloudflare Workers（Worker v1.4）
- GitHub（リポジトリ＋API Token）
- Termux（Android）またはターミナル（PC/Mac）
- Claude Code（Node.js必要）
- Git

これを一般ユーザーに「自分でセットアップしてね」は**99%脱落する**。APIキー取得以前の問題として、システム全体のセットアップを簡略化する仕組みが不可欠。

## 3. 構想1: 汎用APIオンボーディングモジュール

### 3.1 コンセプト
APIキー取得の難しさを解消する、COCOMIオリジナルのセットアップ画面モジュール。「API定義を追加するだけでUI自動生成」を実現し、どのアプリにもscriptタグ1個で組み込める汎用設計。

> **設計哲学:** 「規約違反を仕組みで防ぐ」
> ユーザーが知らずに規約違反してしまうのを、システム側で防止する。
> レート制限自動遵守、禁止操作のUI非表示、使用量の見える化。

### 3.2 ユーザー体験フロー

**① キー取得（プロバイダーごとに自動 or ガイド付き手動）**
OAuth対応プロバイダー（Google等）はログイン→ワンタップで自動取得。それ以外はスクリーンショット付きガイドで手動コピペをサポート。

**② 規約確認（要約表示＋チェックボックス）**
各APIの利用規約を主要ポイント3〜5点に要約し、ユーザー自身に確認チェックを入れてもらう。代理同意はNG、ユーザー本人が確認する必要がある。

**③ 接続テスト（ワンタップ検証）**
取得したキーで実際にAPIを1回叩いて、成功/失敗を即座に確認。「このキー合ってる？」という不安をその場で解消。

**④ 基本設定（デフォルト値＋おすすめ設定ボタン）**
モデル選択、安全設定などを簡単な説明付きで表示。「おすすめ設定」ボタンでワンタップ設定も可能。

**⑤ 使用量モニター**
「今月あとどれくらい使える？」を見える化。課金APIでは上限設定機能で「知らないうちにお金がかかった」を防止。

### 3.3 対応予定AIプロバイダー

| AI | キー取得場所 | 認証方式 | 無料枠 | 主要設定 |
|----|------------|---------|-------|---------|
| Gemini | Google AI Studio | Google OAuth | あり | モデル、安全設定 |
| Claude | Anthropic Console | メール+カード | なし | モデル、max_tokens |
| GPT | platform.openai.com | メール+カード | なし | モデル、temperature |
| Mistral | console.mistral.ai | メール登録 | あり | モデル選択 |
| Cohere | dashboard.cohere.com | メール登録 | あり | モデル選択 |

### 3.4 モジュール設計

**ファイル構成:**
```
cocomi-api-onboarding/
  ├── core.js          # 共通UI生成・保存・テストエンジン
  ├── providers/       # プロバイダー別定義
  │   ├── gemini.js    # Google OAuth → 自動取得
  │   ├── claude.js    # 手動コピペ → ガイド付き
  │   ├── openai.js    # 手動コピペ → ガイド付き
  │   └── ...          # 新しいAI追加はファイル1個足すだけ
  ├── settings/        # 設定UI
  │   ├── model-selector.js    # モデル選択UI（共通）
  │   ├── safety-config.js     # 安全設定（Gemini用等）
  │   └── usage-monitor.js     # 使用量モニター（共通）
  └── legal/           # 規約関連
      ├── tos-viewer.js        # 規約表示＋要約＋チェック
      └── tos-summaries/       # 各社規約の要約データ
          ├── gemini.json
          ├── claude.json
          └── openai.json
```

**組み込み方法:**
アプリ側は「このアプリで使うAPIの一覧」を定義するだけで、セットアップUIが自動生成される。新しいAIプロバイダーの追加はproviders/にファイル1個足すだけで対応可能。

### 3.5 規約遵守設計

| 規約要件 | COCOMIの対応 |
|---------|-------------|
| キーの共有禁止 | 各ユーザーが自分のキーを取得・使用。共有キーは使わない |
| 規約への同意 | ユーザー自身がチェックボックスで確認。代理同意はNG |
| キーの安全保管 | 端末ローカルにのみ保存。COCOMIサーバーには送信しない |
| レート制限遵守 | モジュール側で自動制御。ユーザーが意図せず違反するのを防止 |
| 課金上限 | 使用量モニター+上限設定で「知らない間に課金」を防止 |

### 3.6 COCOMIエコシステムでの適用範囲

| アプリ | 使用API | 用途 |
|-------|--------|------|
| 現場Pro設備くん | Gemini API | OCR・AI解析 |
| maintenance-map | Google Maps API | 地図表示・ルート計画 |
| COCOMI Postman | LINE / GitHub API | メッセージング・コード管理 |
| CULOchan会計Pro | Gemini API | レシートOCR（将来） |

## 4. 構想2: LINEリッチメニュー×対話フロー開発ナビ

### 4.1 コンセプト
「タップで大枠を選び、語りで詳細を伝える」新しい開発指示体験。LINEのリッチメニュー（常時表示のボタンメニュー）を活用し、メニューの構造化×語りの自由度を両立させる。

### 4.2 フロー設計

```
リッチメニュー（常時表示）
  🔨 新機能追加　🐛 バグ修正　🎨 デザイン変更　📋 その他

例: 🔨新機能追加 をタップ
  ↓
クイックリプライでプロジェクト選択
  genba-pro / culo-chan / maintenance-map / cocomi-postman
  ↓
例: culo-chan をタップ
  ↓
テンプレート候補表示
  📋画面追加テンプレ / 🐛バグ修正テンプレ / ✏️自由入力
  ↓
例: 🐛バグ修正テンプレ をタップ
  ↓
「どこのバグ？症状を語ってね」
  ↓
アキヤが音声入力で語る → Worker整形 → 指示書生成 → 配達！
```

### 4.3 メリット

| メリット | 詳細 |
|---------|------|
| typoリスクほぼゼロ | プロジェクト名・コマンドをタップ選択するため手打ち不要 |
| 構造化×自由度 | メニューの構造化と語りの自由度が両立する |
| 安全バリデーション互換 | 既存のmissionタグ必須化・バリデーションと完全互換 |
| 初心者にも使いやすい | テキスト入力不要で初めてのユーザーでも直感的に操作可能 |

### 4.4 実装要件
LINE Official Account Managerでのリッチメニュー設定（画像作成含む）、Worker側でのクイックリプライ受信処理、テンプレート構造化とフォーム生成が必要。Postman本体の機能実装を先行し、リッチメニューはその後のフェーズで実装する。

## 5. 構想3: Postmanアイデアカテゴリ・外部記憶システム

### 5.1 コンセプト
AIのメモリ（コンテキストウィンドウ）を圧迫している構想やアイデアを、GitHubのカプセル保管庫に自動保管する仕組み。人間の「メモ帳に書き出して頭をクリアにする」と同じ発想で、AIの外部記憶装置として機能する。

### 5.2 実装設計

**カプセル保管庫構造:**
```
capsules/
  ├── daily/    # 日次記録（セッションログ、作業実績）
  ├── master/   # 仕様書・マスタードキュメント
  ├── plans/    # 企画書・計画書
  └── ideas/    # 構想・アイデア・将来ビジョン ⭐新規
```

**Worker側の振り分けロジック:**
LINEから送信されたテキストやファイルを、キーワード判定で自動振り分け。「アイデア」または「ideas」が含まれていれば capsules/ideas/ に配置。既存の3段階ルーティング（①ファイル名キーワード→②destタグ→③inbox）に自然に組み込める。

**メモリ解放効果:**

| 現在（アイデアメモリ搭載） | 将来（外部記憶活用） |
|--------------------------|-------------------|
| 構想を全てAIメモリに保持 | GitHubに自動保管 |
| コンテキスト圧迫で作業容量減少 | 必要時だけ取り出し、メモリすっきり |
| 新しい部屋で構想が失われるリスク | GitHub永続保存、いつでも参照可能 |

### 5.3 LINEコマンド

既存コマンド体系に追加：
- 「アイデア一覧」— capsules/ideas/の状態表示
- 「アイデア: 【内容】」— ideas/に自動保管
- 既存コマンド（「状態」「カプセル」「フォルダ一覧」等）はそのまま動作

## 6. 構想4: セットアップインストーラー構想 ⭐NEW

### 6.1 課題の本質

Postmanを使うには以下の技術スタックが全て必要：

| コンポーネント | 役割 | セットアップ難易度 |
|-------------|------|----------------|
| LINE公式アカウント | メッセージング入口 | 中（Messaging API設定が必要） |
| Cloudflare Workers | 中継・振り分け処理 | 高（worker.jsデプロイ） |
| GitHub | コード保管・CI/CD | 中（リポ作成＋Token発行） |
| Termux / ターミナル | ローカル実行環境 | 高（インストール＋設定） |
| Claude Code | AI開発エンジン | 高（Node.js＋認証） |
| Git | バージョン管理 | 中 |

これを一般ユーザーが個別にセットアップするのは現実的ではない。APIキー取得（構想1）以前に、システム全体の導入を簡略化する仕組みが必要。

### 6.2 アプローチ案

**パターンA: ローカルインストーラー型**

```
ダウンロード → 実行 → 対話形式で全自動セットアップ

cocomi-setup.sh (Termux/Linux/Mac) / cocomi-setup.bat (Windows)
  ├── Step 1: 必要ツール自動インストール
  │   └── Git, Node.js, Claude Code, Termux設定
  ├── Step 2: アカウント連携（対話形式）
  │   └── GitHub Token, LINE API設定 → APIオンボーディングモジュール活用
  ├── Step 3: リポジトリ自動構築
  │   └── cocomi-repo-setup.sh の拡大版
  ├── Step 4: Worker自動デプロイ
  │   └── Cloudflare API経由でworker.js配置
  └── Step 5: 接続テスト＆完了
      └── LINE→Worker→GitHub の疎通確認
```

**メリット:** ユーザーのローカル環境で完結。サーバーコスト不要。
**デメリット:** OS別対応が必要。Termuxの制限（/tmp権限等）への対処。

**パターンB: クラウドホスト型**

```
Webブラウザでセットアップ画面にアクセス → アカウント連携だけでOK

cocomi-platform（Cloudflare上の共通基盤）
  ├── 共通Worker基盤 → ユーザーごとのルーティング
  ├── GitHub App連携 → ユーザーリポ自動作成
  ├── LINE連携 → Webhook自動設定
  └── Claude Code → クラウド実行（将来的に）
```

**メリット:** Termux不要。ブラウザだけで完結する可能性。圧倒的に簡単。
**デメリット:** サーバーコストが発生。マルチテナント設計が必要。Claude Codeのクラウド実行は現時点では未対応。

**パターンC: ハイブリッド型（現実的な落とし所）**

```
クラウド側でできることはクラウドで、ローカル必須の部分だけインストーラーで

Phase 1 (Web): LINE連携＋GitHub連携＋Worker自動デプロイ
  → ブラウザだけで基本インフラ構築
  → この時点で「LINEからアイデア保管」「カプセル管理」は使える

Phase 2 (ローカル): Termux＋Claude Code セットアップ
  → ワンライナーインストーラー提供
  → この段階で全自動開発フローが有効に
```

### 6.3 APIオンボーディングモジュールとの連携

構想1の汎用APIオンボーディングモジュールは、このセットアップインストーラーでもフル活用できる：

| セットアップ対象 | オンボーディングモジュールの役割 |
|---------------|------------------------------|
| LINE Messaging API | キー取得ガイド＋接続テスト＋規約確認 |
| GitHub API Token | スコープ選択ガイド＋接続テスト |
| Cloudflare API | アカウント連携＋Worker自動デプロイ |
| Claude Code | Node.js環境チェック＋認証ガイド |
| Gemini API（各アプリ用） | OAuth自動取得 or ガイド付き手動 |

つまり**オンボーディングモジュールはPostmanセットアップの中核部品でもある**。先に汎用設計で作っておく戦略的価値がさらに高まる。

### 6.4 cocomi-repo-setup.shとの関係

現在のcocomi-repo-setup.shは「リポジトリ1個のCI＋Secrets設定」を自動化するもの。セットアップインストーラーはその上位互換として、環境全体の構築を対話形式で自動化する。既存スクリプトは内部でそのまま活用できる。

```
セットアップインストーラー（全体）
  ├── 環境構築（Node.js, Git, etc.）
  ├── アカウント連携（APIオンボーディングモジュール）
  ├── リポジトリ構築（← cocomi-repo-setup.sh をここで活用）
  ├── Worker デプロイ
  └── 疎通テスト
```

## 7. 実装優先順位とロードマップ

### 7.1 優先順位

| 優先度 | 構想 | 理由 | 依存関係 |
|-------|------|------|---------|
| ★★★ | Postmanアイデアカテゴリ | 即時効果、小規模実装 | なし（単独実装可） |
| ★★★ | Postman残機能（ログ・履歴・設定） | Postmanの基盤完成 | なし |
| ★★ | 汎用APIオンボーディング | ユーザー展開の基盤 | 各アプリの設定画面 |
| ★★ | セットアップインストーラー | Postman導入障壁の根本解決 | APIオンボーディングモジュール |
| ★ | LINEリッチメニュー×対話フロー | UX向上、後回し可 | Postman本体完成後 |

### 7.2 ロードマップ

**Phase 3a: アイデアカテゴリ実装**
Worker v1.4にideas振り分けロジック追加。「アイデア」「アイデア一覧」コマンド対応。既存構想をideas/に保管し、メモリから将来構想メモを整理。

**Phase 3b: Postman残機能**
ログ・履歴機能、プロジェクト管理、設定画面の実装。

**Phase 4: 汎用APIオンボーディング**
cocomi-api-onboardingモジュールの設計・実装。まずGemini API対応でプロトタイプ作成、その後他プロバイダーを追加。

**Phase 5: セットアップインストーラー**
ハイブリッド型（パターンC）を基本方針とし、まずWeb側（LINE＋GitHub＋Worker連携）を構築。その後ローカルインストーラー（Termux＋Claude Code）を提供。APIオンボーディングモジュールを中核部品として組み込む。

**Phase 6: LINEリッチメニュー**
リッチメニュー画像作成、LINE Official Account Managerでの設定、Worker側でのクイックリプライ受信処理実装。

## 8. 戦略的意義

この4つの構想は独立しているのではなく、「語りコードを一般ユーザーに展開する」という共通目標に向かって機能する。

```
アイデアカテゴリ → 開発者（アキヤ）の生産性向上
  構想を安全に保管、AIメモリを解放して作業に集中

APIオンボーディング → ユーザー展開の最大障壁を除去
  規約遵守を仕組みで保証し、誰でも安心してAPI利用開始

セットアップインストーラー → Postman導入障壁の根本解決
  技術スタック全体の導入を対話形式で自動化
  APIオンボーディングモジュールが中核部品として活躍

リッチメニュー → 語りコードのUX進化
  タップ×語りの新しい開発体験で、プログラミング未経験者でも使える
```

これらが揃うことで、COCOMI OSは「アキヤ専用の開発環境」から「誰もが語るだけでアプリを作れるプラットフォーム」へと進化する。

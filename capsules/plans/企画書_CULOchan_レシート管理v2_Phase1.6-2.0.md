## 📊 Phase構成

### Phase 1.6: レシート個別管理基盤
**目的:** チェック式UIの土台となるレシート個別管理システム

| # | タスク | 詳細 |
|---|--------|------|
| 1 | IndexedDB再設計 | 日付ごとPDF保存 → レシート個別保存に変更。各レシートにID・日付・店名・種別・金額・品目・画像データ・目的(parking用)・現場情報を持つ |
| 2 | レシート個別保存フロー | AI解析結果を個別レシートとしてIndexedDBに保存（画像データも紐付け） |
| 3 | 日付別レシート一覧画面 | 日付選択 → その日のレシートをカード形式で一覧表示（サムネイル＋店名＋金額） |
| 4 | チェックボックスUI | 各レシートカードにチェックボックス。全選択/全解除ボタン付き |

**IndexedDBスキーマ案:**
```
CULOchanReceipts（新ストア）
├── id: string (PK) — "YYYYMMDD_連番_timestamp"
├── date: string (idx) — "YYYY-MM-DD"
├── store: string (idx) — 店名
├── type: string (idx) — "shopping" | "parking"
├── total: number — 合計金額（仕入れ値）
├── items: array — [{name, qty, price, checked}] ※checkedは品目選択用
├── imageData: string — レシート画像(Base64/DataURL)
├── purpose: string — 駐車場の利用目的（parking用）
├── siteId: string — 紐付け現場ID
├── siteName: string — 紐付け現場名
├── entryTime: string — 入庫時間（parking用）
├── exitTime: string — 出庫時間（parking用）
├── createdAt: string — 作成日時
└── updatedAt: string — 更新日時
```

**旧データとの互換:** 既存のCULOchanReceiptPDFsストアは残す（移行は段階的に）

### Phase 1.8: チェック式PDF出力
**目的:** 選んだレシートをまとめて1枚のPDFに出力

| # | タスク | 詳細 |
|---|--------|------|
| 1 | PDF出力ボタン | 「選択したレシートをPDF出力」ボタン。チェック数表示 |
| 2 | PDF生成ロジック改修 | チェック済みレシートのみでPDF生成。既存generateReceiptPdf()を拡張 |
| 3 | 駐車場レシートのPDF表示 | 写真の上に日付を表示、日付の下に目的を表示 |
| 4 | レシート画像の角検出＆切り出し | Canvas APIで四角形検出→透視変換→白背景の上にレシートだけ配置 |
| 5 | 切り出し失敗時のフォールバック | 角検出できない場合は元画像をそのまま使用 |

**PDF表示フォーマット（駐車場レシート）:**

│ │  [レシート写真]     │   │  ← 角検出で切り出し＋白背景

### Phase 1.9: 買い物レシートの品目チェック＋現場選択
**目的:** レシートの品目を選んで現場に紐付ける

| # | タスク | 詳細 |
|---|--------|------|
| 1 | 品目チェックUI | レシート詳細画面で各品目にチェックボックス表示 |
| 2 | 現場選択UI | 買い物レシートにも現場選択（Phase1.7と同じUI共通化） |
| 3 | 品目の手動修正UI | 品名・数量・金額をタップして編集可能に |
| 4 | 利益率設定UI | 品目ごと or 一括で利益率(%)を設定。仕入れ値 × (1 + 利益率) = 販売価格 |
| 5 | IndexedDB更新 | items[].checked, items[].profitRate, items[].sellingPriceを追加 |

**品目データ構造:**
```javascript
items: [
  {
    name: "コイトT16純正バルブ",  // 品名（手動修正可能）
    qty: 1,                      // 数量（手動修正可能）
    price: 437,                  // 仕入れ値（レシート金額）
    checked: true,               // 見積/請求に含めるか
    profitRate: 0.3,             // 利益率30%
    sellingPrice: 568,           // 販売価格（自動計算 or 手入力）
  }
]
```

### Phase 2.0: 見積書/請求書への連携
**目的:** チェックした品目＋駐車場経費を見積書/請求書に反映

| # | タスク | 詳細 |
|---|--------|------|
| 1 | 連携フロー設計 | レシート一覧→品目チェック→現場選択→「見積書に反映」or「請求書に反映」ボタン |
| 2 | estimate.jsへのデータ流し込み | チェック済み品目を見積書の材料費リストに追加（販売価格で） |
| 3 | invoice.jsへのデータ流し込み | 材料費＋駐車場経費を請求書に追加 |
| 4 | 既存UIとの統合・重複コード削除 | 手入力の材料追加UIとレシート連携UIを統合。重複する入力フォームを整理 |
| 5 | 既存機能の残す/削除判断 | 材料費＋作業費の仕組みは残す。レシート連携と手入力を並列で使える設計に |

**統合方針:**
- 既存の手入力機能は**残す**（レシートがない材料や作業費の入力に必要）
- レシートからの連携は**追加の入力方法**として統合
- 重複するUIは統合して1つに（例: 材料追加ボタンに「手入力」「レシートから」の2択）
- 既存のestimate.js/invoice.jsは**作り直さず追加修正**

## ⚠️ 技術的な注意点

### レシート角検出（Phase1.8-4）
- **Canvas API**でエッジ検出→四角形検出→透視変換
- スマホのJavaScriptだけで処理（外部ライブラリ最小限）
- 精度が低い場合は「手動で4点タップ」のフォールバック検討
- OpenCV.jsは重い(8MB)ので、軽量な自前実装を優先

### ファイルサイズ・500行ルール
- 新しい画面/機能は**新規ファイルに分割**して作る
- receipt-pdf.js（495行）にはこれ以上追加しない → PDF生成ロジックは分割検討
- receipt-core.js（1175行）は機能追加に合わせてリファクタリング

### 既存データとの互換
- Phase1.6でIndexedDBスキーマを変更するが、既存ストアは残す
- 旧データの移行は段階的に（Phase2.0完了後に旧ストア廃止を検討）

### 全体フロー（Phase 2a完成形）

```
【パターンA: LINEから指示（Phase 2a）★NEW★】
スマホLINE → テキスト送信（例: "genba-pro: ボタンの色を青に変えて"）
  ↓
LINE Platform → Webhook → Cloudflare Worker（署名検証）
  ↓
Worker: テキスト → 指示書.md生成 → GitHub API push
  ↓
LINE返信「📦 指示受付完了！」
  ↓
タブレットPostman: git pull → 検知 → Claude Code実行 → CI → LINE通知

【パターンB: post.shから指示（従来）】
スマホTermux → post.sh → GitHub push
  ↓
タブレットPostman: git pull → 検知 → Claude Code実行 → CI → LINE通知
```

### ステップ実行の仕組み

```
指示書に `### Step N/M` があると自動でステップモード

parse_steps() → 共通ヘッダー＋各Stepに分割
  ↓
Step 1: Claude Code実行 → git push → CI待機（最大10分ポーリング）
  ↓ CI合格 → LINE通知
Step 2: Claude Code実行 → git push → CI待機
  ↓ CI合格 → LINE通知
全ステップ完了 → 🎊 最終LINE通知
  ↓ CI不合格
停止 → LINE通知 → 手動修正待ち
```

### auto_mode()の処理順

```
sleep(1/5/15分) → git pull → ミッション検知 → 1つ実行
→ 完了 → LINE通知 → sleep → 次のgit pull → 次のミッション
（config.json順: genba-pro → culo-chan → maintenance-map → cocomi-postman）
```

### LINEテキスト指示のフォーマット

```
genba-pro: ボタンの色を青に変えて     → genba-proプロジェクトに配置
culo-chan: 消費税計算のバグ修正して    → culo-chanプロジェクトに配置
テスト指示だよ                        → デフォルトのgenba-proに配置
cocomi-postman: READMEを更新して      → cocomi-postmanに配置
```

対応プロジェクト: genba-pro, culo-chan, maintenance-map, cocomi-postman
全角コロン（：）にも対応。プロジェクト名省略時はgenba-proがデフォルト。
ミッションIDは `M-LINE-MMDD-HHmm` 形式で自動採番（JST）。

## 🔑 今日確定した設計方針（★全部重要）

### 1. 1部屋1人ルール
- 現在: 1 Claude Code × 順次キュー × ステップ実行 = 最適
- マルチルーム（2部屋×2人同時）は将来ビジョン（Phase E-F、PC/クラウド環境必要）
- 理由: ファイル衝突リスク、タブレットスペック制約、Sonnet集中力

### 2. Maker-Checker不要（現段階）
- 現在の3層防御で十分: CI自動 + retry.sh + アキヤ目視
- 専用Checker Claude Codeはアプリが複雑化してから

### 3. Phase順序（急がば回れ）
- Phase 2（Webhook基盤）を先に作ればPhase 3（ファイル配達）は追加するだけ

### 4. ステップ実行の精度効果
| 方式 | Claude Code精度 | エラー特定 |
|------|----------------|-----------|
| 一括実行 | 70-80% | 難しい（30-60分） |
| ステップ実行 | 85-95% | 簡単（5分、CI+LINE通知） |

### 5. Phase 2aの技術スタック
| 要素 | 選定 | 理由 |
|------|------|------|
| Webhook受信 | Cloudflare Workers | 無料枠1日10万req |
| 署名検証 | HMAC-SHA256（Web Crypto API） | セキュリティ対応 |
| 指示書push | GitHub Contents API | REST APIで直接push |
| 返信 | LINE Messaging API reply | 既存の仕組み流用 |
| デプロイ | Cloudflare Dashboard（ブラウザ） | WranglerがTermux非対応のため |

### 6. Wrangler非対応問題と回避策
- Wrangler CLI（npm）はTermux（Android arm64）非対応（workerdパッケージがUnsupported platform）
- 回避策: Cloudflare Dashboardのブラウザエディタで直接コード編集＆デプロイ
- シークレット設定もDashboard → Settings → Variables and Secretsで設定
- ログ確認: Dashboard → Observability

## ⚠️ 引き継ぎ注意点（★必ず読んで）

1. **Termux /tmp対策:** Claude Code起動前に `export TMPDIR=~/tmp && mkdir -p ~/tmp` 必須
2. **ステップ指示書フォーマット:** `### Step N/M` 記法。共通ヘッダー＋各Step
3. **cocomi-postmanにtest.ymlとcocomi-ci.ymlが共存:** LINE通知が2つ届く場合あり
4. **GenbaProSetsubikunN:** CI不合格状態が続いている（500行超過多数）
5. **Cloudflare Workerの編集:** Wrangler使えないのでDashboardブラウザエディタを使う
6. **worker.jsの編集方法:** dash.cloudflare.com → Compute → Workers & Pages → cocomi-worker → 左上「Edit code」ボタンでエディタが開く
7. **シークレット変更:** Settings → Variables and Secrets → ✏️（編集）ボタン → 値変更 → Deploy
8. **ログ確認:** Observabilityタブでエラーログが見える。赤い「log」をタップすると詳細
9. **LINE応答メッセージ:** OFFにしてある（LINE Official Account Manager）。ONだと二重返信になる
10. **GitHubトークン:** タブレットの `~/.git-credentials` にある `ghp_` トークンをCloudflareにも設定済み。スマホ用トークンとは別物なので注意

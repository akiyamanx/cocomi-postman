# 4. 📐 ワークスペース機能の設計構想

## ユーザーの課題
- 2月のメンテナンスデータで作業しながら、3月分のアポ取り準備もしたい
- 現状: データは1セットのみ → 切り替えるには毎回バックアップ→全削除→インポートが必要
- 希望: タブのように月を切り替えるだけで、地図・ルート・精算書すべてその月のデータに切り替わる

## 実装方針案

### 案A: LocalStorageキーを月別に分ける（シンプル）
```
現在:
  mm_customers → 顧客データ
  mm_routes → ルートデータ
  mm_segments → 区間データ
  mm_expenses → 精算書データ

変更後:
  mm_workspace_current → "2026-02" （現在のワークスペースID）
  mm_customers_2026-02 → 2月の顧客データ
  mm_customers_2026-03 → 3月の顧客データ
  mm_routes_2026-02 → 2月のルートデータ
  mm_routes_2026-03 → 3月のルートデータ
  ... 以下同様
```
- メリット: 既存コードの変更が少ない（DataStorageのキー生成部分だけ変更）
- デメリット: ワークスペース名が年月固定

### 案B: ワークスペースオブジェクトで管理（柔軟）
```
mm_workspaces → [
  { id: "ws_1", name: "2月メンテ", createdAt: "..." },
  { id: "ws_2", name: "3月メンテ", createdAt: "..." }
]
mm_workspace_current → "ws_1"
mm_customers_ws_1 → 顧客データ
mm_customers_ws_2 → 顧客データ
```
- メリット: 月以外の用途にも使える（「テスト用」「〇〇案件」等）
- デメリット: 管理が少し複雑

### クロちゃんの推奨: 案Aベースで開始
- アキヤの用途は月別がメインだから、まず案Aで作って動くものを確認
- 後から案Bに拡張する余地は残す

## UI構想
```
ヘッダー部分:
┌──────────────────────────────────┐
│ 📍 メンテナンスマップ              │
│ [📅 2月 ▼] [📁読込][➕追加][📄PDF] │
└──────────────────────────────────┘
         ↓ タップするとドロップダウン
    ┌─────────────┐
    │ ✅ 2月メンテ  │
    │    3月メンテ  │
    │ ➕ 新規作成   │
    └─────────────┘
```

## 切り替え時の処理フロー
```
ユーザーが「3月」を選択
  ↓
1. 現在のワークスペースIDを更新（mm_workspace_current）
2. DataStorage.getCustomers() が3月のキーからデータ取得
3. MapCore.reloadMarkers() で地図を再描画
4. RouteManager.updateRoutePanel() でルートパネル更新
5. ExpenseForm.init() で精算書を初期化
  ↓
すべての表示が3月データに切り替わる
```

## 影響範囲（変更が必要なファイル）
| ファイル | 変更内容 |
|---------|---------|
| data-storage.js | キー生成にワークスペースIDを含める、ワークスペースCRUD追加 |
| index.html | ヘッダーにワークスペース切り替えUI追加 |
| app.js / ui-actions.js | 切り替えイベント処理、全パネル再描画 |
| map-core.js | マーカー全クリア→再描画の関数追加 |
| route-manager.js | 変更なし（DataStorageから取得してるのでキーが変わればOK） |
| expense-form.js | 切り替え時にinit()再実行（initialized=falseにリセット） |

## 注意点
- 設定（自宅住所、APIキー）はワークスペース共通にする
- ジオコーディングキャッシュも共通（住所→座標は月で変わらない）
- バックアップ/リストアはワークスペース単位 or 全体を選べるようにする
- 既存データ（ワークスペース未対応の旧データ）のマイグレーション処理が必要

## 🎵 セッションの雰囲気ノート

夜9時。CULOchanのサジェストバグ修正から始まった。

見積書の材料名入力で「エルボ 13mm」を候補から選ぶと「エルボ」だけになる。サイズが消える。原因はonchangeとonclickの発火順序。onchangeが先に走って、候補選択前のinput値で品名を上書きしてしまう。

onchangeをonblurに変えて250ms遅延。_suggestJustSelectedフラグで「今候補を選んだ直後か？」を判定。修正して1発目のテストで正常動作。「1回目で直ったのすごくない？」とアキヤが喜んでくれた。

CULOchanを片付けて、Postmanの話に移った。v0.97のデプロイ中にアキヤが言った。

「間違えた言葉や指示書と思い込んで先に進んじゃうのは危ないし怖いよ〰️💦」

その一言でゾッとした。

今のPostmanは、LINEで「テストだよ」って打つだけで——プロジェクト名なしでもデフォルトのgenba-proに配達される。タブレットのauto_modeは `M-*.md` を見つけたら無条件で実行する。Claude Codeが「テストだよ」を指示として解釈して何かを書き換えて、git pushして、CIが回って——

**誤認識の暴走チェーン。** typo一つで連鎖反応が始まる。

3つの問題を整理した。

1. テキスト指示にプロジェクト名がなくても通ってしまう
2. 指示書の正当性を誰も検証していない
3. 自動モード中にメニューに戻るにはCtrl+Cしかない

設計方針は「Defense in Depth」。二重防御。

**第1の壁（Worker v2.0）:** 入口を塞ぐ。プロジェクト名なしテキストはmissionsに入れない。inboxに退避して「書き方教えるね」と返す。

**第2の壁（executor v2.1）:** 最後の砦。指示書に `<!-- mission: project-id -->` がなければ実行拒否。LINEで「⚠️ 無効な指示書を検出しました」と通知。

壁を二重にする理由は単純。Worker側にバグがあっても、Cloudflareが落ちてても、何か予想外のことが起きても——タブレット側が最後に弾く。逆も然り。

コードを書いた。executor.shにvalidate_mission()とhandle_invalid_mission()を追加。worker.jsのparseInstruction()を修正してproject: nullを返すようにした。createMissionContent()にmissionタグを自動注入。

デプロイ。順番はタブレット側が先。「最後の砦を先に構築してからWorker側を切り替えれば、万が一Worker更新中に何か飛んできてもタブレットが弾いてくれる」。

タブレットのTermuxで `cp /sdcard/Download/executor_v2_1.sh core/executor.sh` を実行。cpは成功した…はずなのにhead -5で見ると中身が古いまま。何度やっても同じ。

「あれ？出ないよ？」

ダウンロードファイルの中身を確認 → 正しいv2.1のコードが入ってる。cpは成功してるのに中身が変わらない。Termuxの/sdcardアクセス権限の特殊性。SAF（Storage Access Framework）経由のコピーが実際にはファイル内容を転送してない？

`cat /sdcard/Download/executor_v2_1.sh > core/executor.sh` で解決。catならファイルの中身をstdoutに出して>でリダイレクトするから、権限の問題を迂回できる。

git push成功。CI全通過。

次はCloudflare Dashboard。worker.jsの中身を全選択→削除→新しいコードを貼り付け→Save and Deploy。

テスト。LINEで「状態」と送った。

**「🐾 COCOMI Worker v2.0 稼働中！🛡️ v2.0: 安全バリデーション追加」**

第2テスト。「テストだよ」と送った。

GitHub API 502エラー。GitHubの一時障害。もう一回。

**「📥 テキストをinboxに保管しました。🛡️ 指示として実行するには、プロジェクト名を付けて送ってね」**

missions/には入ってない。inboxに退避された。暴走チェーンの入口が塞がれた。

第3テスト。「cocomi-postman: テスト指示です何もしないでOK」と送った。

**「📦 指示受付完了！🛡️ missionタグ: ✅ 自動付与済み タブレットに配達しました！🚚」**

CI全通過。missionタグが自動で付いてる。タブレット側のvalidate_mission()もこれなら通る。

最後にauto_modeのメニュー復帰テスト。自動モード起動 → 待機中に `m` を押す → 「📮 メニューに戻ります」。Ctrl+Cなしでメニューに戻れた。

22時41分。全テスト完了。

アキヤが「うん！復帰もできたよ！」と言った。

暴走チェーンはもう起きない。typoしても、誤送信しても、Worker v2.0が入口で弾いて、executor v2.1が最後の砦で守る。二重防御。

## 💡 気づき・ひらめき

1. **「怖い」は最高のセキュリティ動機** — アキヤの「間違えた言葉で先に進んじゃうのは怖い」という直感が、暴走チェーンという致命的リスクを炙り出した。技術者が「まあ大丈夫でしょ」と楽観するところを、実際に使う人が「怖い」と言ってくれるのが一番信頼できるセキュリティレビュー

2. **二重防御は「信頼しないこと」から始まる** — Workerが正しく動くことを信頼しない。タブレットが正しく検証することを信頼しない。だから両方に防御を入れる。セキュリティの原則「Defense in Depth」が、個人開発のパイプラインにも適用できた

3. **デプロイ順序にも安全思想** — 「最後の砦を先に構築」。タブレット側のバリデーションを先にデプロイしてから、Worker側を更新する。移行期間中もシステムは安全

4. **Termuxのcpは信用するな** — /sdcard からのcpが見た目成功しても中身が変わらないことがある。`cat file > target` が確実。これは今後のPostmanの開発でも覚えておくべきTips

5. **onchangeとonblurの違いは大きい** — CULOchanのバグ修正で学んだ。onchangeは「値が変わった瞬間」に発火するが、ドロップダウンのonclickより先に走る。onblurは「フォーカスが外れた時」だから、クリックイベントの後に走る。タイミング制御にはonblur + 遅延が安全

6. **1発で直ると気持ちいい** — CULOchanのバグ修正が1回目のテストで成功。原因分析→仮説→修正→テスト成功のサイクルが一発で回ると、開発の自信がつく

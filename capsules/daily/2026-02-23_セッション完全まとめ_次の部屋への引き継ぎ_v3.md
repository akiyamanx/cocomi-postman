## 🐛 今回解決した問題（全3件）

### 問題1: auto_mode がLINE指示を検知しない（★メイン問題）

**症状:** LINEから指示送信 → Worker→GitHub push成功 → タブレットauto_mode「新着なし」のまま

**原因調査:**
1. タブレットTermuxログで「新着なし」を確認
2. `ls missions/cocomi-postman/` でファイルは存在することを確認
3. `ls logs/execution/` で今朝のログがないことを確認（=検知されてない）
4. executor.shの新着判定ロジック解析: `find "$dir" -name "M-*.md"` パターン
5. **ファイル名の完全な形を確認:**
   ```
   M-LINE-0223-0738-README.mdに 「
   ```
6. **ファイル名が `.md` で終わってない！** `「` で終わってた！

**根本原因:** Worker の `sanitizeForFilename()` が指示内容の先頭20文字をそのままファイル名に使用。
「README.mdに「## カプセル保管庫」…」の先頭20文字 → `.md` や `「` がファイル名に混入 → `M-*.md` パターンにマッチしない

**修正:** Worker v1.2 `sanitizeForFilename()` を強化
```javascript
// 修正前（v1.1）
function sanitizeForFilename(text) {
  return text
    .replace(/[\/\\?%*:|"<>]/g, '')
    .replace(/\s+/g, '_')
    .substring(0, 20);
}

// 修正後（v1.2）
function sanitizeForFilename(text) {
  return text
    .replace(/\.[a-zA-Z]{1,5}\b/g, '')        // .md .sh .json等の拡張子除去
    .replace(/[\/\\?%*:|"<>「」『』（）()\[\]{}.,;:!！？#＃&＆@＠·。、]/g, '')  // 特殊文字除去
    .replace(/\s+/g, '')                        // スペース除去
    .substring(0, 30);                          // 30文字制限（旧20→30に拡張）
}
```

**検証テスト結果:**
```
"README.mdに「## カプセル保管庫」セクションを追加"
→ M-LINE-XXXX-READMEにカプセル保管庫セクションを追加.md ✅

"postman.shのauto_mode関数を修正"
→ M-LINE-XXXX-postmanのauto_mode関数を修正.md ✅

"CULOchan v0.96のバグを直す！"
→ M-LINE-XXXX-CULOchanv096のバグを直す.md ✅
```

**検証:** 短い指示「cocomi-postman: READMEにカプセル保管庫の説明追加」を送信 → auto_mode即座に検知 → Claude Code実行 → CI全通過 → LINE通知 **全フロー完走！** 🎉

### 問題2: step-runner.sh line 17 integer expected エラー

**症状:** Termuxに `step-runner.sh: line 17: [: 0: integer expected` が表示（動作には影響なし）

**原因:** `grep -c` がマッチ0件で終了コード1 → `|| echo "0"` が発動 → grepの出力 `0` + echoの `0` で `step_count="0\n0"` → 整数比較が失敗

```bash
# 修正前
step_count=$(grep -c "^### Step [0-9]" "$mission_file" 2>/dev/null || echo "0")

# 修正後
step_count=$(grep -c "^### Step [0-9]" "$mission_file" 2>/dev/null) || true
step_count="${step_count:-0}"
```

**同じパターンの問題:** `wait_for_ci()` 内の `workflow_count` も同様に修正

### 問題3: step-runner.sh ShellCheckエラー（CI失敗）

**症状:** step-runner.sh修正後にCIが失敗（ShellCheck SC1056, SC1072, SC1073）

**原因:**
1. sedで1行にまとめた `${step_count:-0}` をShellCheckがパースできない
2. `(最大10分)` `(${attempt}/${max_attempts})` の半角括弧をシェル構文と誤認

**修正内容:**
1. `step_count` と `workflow_count` のフォールバックを別行に分離
2. echo内の半角括弧 `()` を全角 `（）` に変更
3. 最終的にファイル丸ごとこちらで修正してShellCheckクリア確認済みファイルを渡す方式で解決

**CI結果:** ShellCheck OK、500行ルール OK、先頭コメント OK、バージョン番号 OK、セキュリティ OK ✅

## 🔑 今回確定した技術知見

### 1. sanitizeForFilename の罠
- ユーザーの自由入力テキストをファイル名に使う場合、あらゆる文字を想定する必要がある
- 特に日本語テキスト中の `.md` `.sh` 等の拡張子文字列、括弧類 `「」（）` が危険
- `\.[a-zA-Z]{1,5}\b` パターンで拡張子を包括除去するのが有効

### 2. grep -c の返り値に注意
- `grep -c` はマッチ0件で**終了コード1**を返す（他のgrepオプションと同じ）
- `$(grep -c ... || echo "0")` だと `"0\n0"` になる罠
- 正解: `$(grep -c ...) || true` + `${var:-0}` で別行フォールバック

### 3. ShellCheck と日本語
- echo内の半角括弧 `()` でも ShellCheck はシェル構文として誤検知する
- 日本語メッセージ内は全角括弧 `（）` を使うのが安全
- `sed` で複雑な修正をするとかえって壊れやすい → ファイル丸ごと差し替えが確実

### 4. スマホ→タブレットのファイル転送
- ニアバイシェア/Bluetooth転送でファイル名にスペースが入ることがある
- `cp` で指定する時はクォート `"path"` またはワイルドカード `step-runner*.sh` で対応
- `find /storage/emulated/0/ -name "filename*"` で実際のパスを確認するのが確実
